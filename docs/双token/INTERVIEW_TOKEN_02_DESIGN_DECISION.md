# 为什么管理端用双Token，用户端不用？（设计决策篇）

## 一、一句话总结

> **管理端追求"持续在线+可控安全"，用户端追求"快速访问+简化体验"**

---

## 二、管理端 vs 用户端 场景对比

| 维度 | 管理端（商家后台） | 用户端（消费者APP） |
|------|-------------------|-------------------|
| **使用时长** | 长时间操作（几小时） | 短时间操作（几分钟） |
| **操作频率** | 高频：接单、处理订单、修改菜品 | 低频：浏览、下单、支付 |
| **安全要求** | 极高（涉及资金、数据） | 较高（涉及支付） |
| **会话中断容忍度** | 低（突然退出影响工作） | 高（重新登录成本低） |
| **设备环境** | 固定办公电脑 | 各种移动设备 |
| **用户群体** | 内部员工，可控 | 外部用户，不可控 |

---

## 三、详细场景分析

### 3.1 管理端为什么必须双Token？

```
场景：餐厅老板小明在后台处理订单

单Token（2小时有效期）：
┌────────────────────────────────────────┐
│ 9:00  小明登录，开始处理订单            │
│ 10:30 正在接单高峰期，Token过期了！     │
│       ❌ 被迫退出，订单处理中断          │
│       ❌ 重新登录，错过3个新订单         │
│ 11:00 刚处理一会儿，又过期了...          │
└────────────────────────────────────────┘

双Token（Access 30分钟 + Refresh 7天）：
┌────────────────────────────────────────┐
│ 9:00  小明登录，开始处理订单            │
│ 10:30 Access Token过期                  │
│       ✅ 后台自动刷新，无感知            │
│       ✅ 继续处理订单，业务不中断         │
│ 18:00 下班，全天无需重新登录             │
└────────────────────────────────────────┘
```

**管理端的核心需求**：
1. **长时间在线**：后台运营是持续工作，不能频繁中断
2. **安全性可控**：员工账号可以后台管理，发现异常可强制下线
3. **操作连贯性**：订单处理、数据统计等操作不能中途断掉

### 3.2 用户端为什么单Token就够了？

```
场景：用户小红点外卖

单Token（2小时有效期）：
┌────────────────────────────────────────┐
│ 12:00 小红打开APP，浏览餐厅             │
│ 12:10 选好菜品，下单支付                │
│ 12:15 订单完成，退出APP                 │
│                                       │
│ Token有效期2小时完全够用                │
│ 整个过程只有15分钟                      │
└────────────────────────────────────────┘

如果用双Token：
┌────────────────────────────────────────┐
│ 增加了复杂度，收益不明显                │
│ 用户用完即走，不需要长期保持登录         │
│ 重新登录成本低（微信一键登录）           │
└────────────────────────────────────────┘
```

**用户端的核心特点**：
1. **用完即走**：外卖场景是短时操作，15分钟完成下单
2. **重新登录简单**：微信授权登录，一键完成
3. **多端登录**：用户可能在手机、平板、网页都登录，双Token管理复杂
4. **安全性相对宽松**：用户只能操作自己的数据，影响范围小

## 四、技术实现差异

### 4.1 管理端双Token

```yaml
sky:
  jwt:
    # 管理端 - 双token
    admin-ttl: 1800000              # Access: 30分钟
    admin-refresh-ttl: 604800000    # Refresh: 7天
```

```java
// 登录返回双token
EmployeeLoginVO vo = EmployeeLoginVO.builder()
    .token(accessToken)           // Access Token
    .refreshToken(refreshToken)   // Refresh Token
    .expiresIn(1800)              // 过期时间提示
    .build();
```

### 4.2 用户端单Token

```yaml
sky:
  jwt:
    # 用户端 - 单token
    user-ttl: 7200000               # 2小时，足够完成一次外卖流程
```

```java
// 登录只返回单token
UserLoginVO vo = UserLoginVO.builder()
    .token(token)                 // 只有一个Token
    .build();
```

## 五、安全风险对比

### 5.1 管理端风险更高

| 风险 | 管理端 | 用户端 |
|------|--------|--------|
| **权限范围** | 查看所有订单、修改菜品、查看营业数据 | 只能看自己的订单 |
| **资金影响** | 可操作退款、修改价格 | 只能支付自己的订单 |
| **数据泄露** | 全量用户数据、营业额 | 仅个人数据 |
| **账号共享** | 员工可能共享账号 | 个人账号，一般不会共享 |

```
管理端Token被盗的后果：
❌ 查看所有用户的个人信息
❌ 查看餐厅营业额数据
❌ 恶意修改菜品价格
❌ 操作订单退款

用户端Token被盗的后果：
⚠️ 查看该用户的订单
⚠️ 用该用户的余额下单
（影响范围仅限该用户）
```

### 5.2 为什么管理端可以用Refresh Token存Redis？

```
管理端特点：
✅ 用户数量少（几十人）
✅ 内部可控，可监控
✅ 需要服务端控制能力（踢人下线）
✅ Redis存储成本可控

用户端如果也用Redis：
❌ 用户数量百万级，Redis压力大
❌ 存储成本高
❌ 服务端维护复杂度高
```

## 六、面试高频问题

### Q1: 为什么用户端不用双Token？

**标准回答**：
> "用户端场景是'用完即走'，一次外卖流程通常15分钟内完成，单Token 2小时有效期完全够用。双Token会增加系统复杂度和存储成本，而用户体验提升不明显。另外用户端使用微信一键登录，重新登录成本很低。"

### Q2: 管理端双Token过期时间怎么定的？

**标准回答**：
> "Access Token 30分钟，保证安全性；Refresh Token 7天，保证一周工作日内无需重新登录。这两个时间是业务导向的：工作日8小时，7天覆盖完整工作周，管理员周末可能也需要看数据。"

### Q3: 如果用户端也要支持长时间登录呢？

**标准回答**：
> "可以延长单Token有效期到7天，或者也实现双Token。但需要考虑：1) 存储成本（百万用户Token存Redis）；2) 多端登录处理（手机、平板、网页）；3) 用户主动退出需要服务端配合吊销。一般C端产品用'长期有效Token+主动退出'更简单。"

### Q4: 管理端和用户端Token泄露分别怎么处理？

**标准回答**：
> "管理端：删除Redis中的Refresh Token，强制用户重新登录；可以立即生效。用户端：只能等Token自然过期，或者修改用户密码使Token失效（如果Token里包含密码签名）。"

## 七、架构设计原则总结

```
┌─────────────────────────────────────────┐
│           架构设计没有银弹               │
├─────────────────────────────────────────┤
│                                         │
│  1. 根据业务场景选择合适的方案           │
│                                         │
│  2. 权衡安全性、用户体验、实现成本        │
│                                         │
│  3. 不同端可以有不同的安全策略           │
│                                         │
│  4. 复杂度只在必要时引入                 │
│                                         │
└─────────────────────────────────────────┘
```

**本项目的设计亮点**：
- ✅ 管理端双Token：保证安全+体验
- ✅ 用户端单Token：简单够用
- ✅ 同一套JWT工具，不同配置
- ✅ 后期可扩展，用户端也可升级双Token
